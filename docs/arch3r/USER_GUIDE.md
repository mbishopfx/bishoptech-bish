# ARCH3R User Guide

This guide explains how a member uses ARCH3R day to day.

## 1. Sign in and choose your workspace

- Sign in with the credentials created by your org admin.
- If you belong to more than one organization, choose the active workspace.
- Your permissions, visible tools, and model access follow the active organization.

## 2. Start work in chat

- Open `Chat` from the left rail.
- Start a new thread or continue a shared thread.
- If a chat is shared, members added to that thread can see the history and continue the conversation.
- Archive a thread when it should stay searchable but leave the main active history.

## 3. Switch models when needed

- Change models from the chat composer.
- ARCH3R keeps the conversation inside one thread and carries recent context forward.
- Best practice: start a new chat for major topic changes or when you need the cleanest full-context result.

## 4. Use shared work patterns

### Shared threads

- Owners can add org members to a thread.
- Participants can continue the conversation.
- Thread owners retain handoff control for local execution flows.

### Huddles

- Open a huddle room from the left rail or from an attached workflow such as a project.
- Huddles are designed for open collaboration, quick voice sync, and notes.
- You can move around the platform while staying active in a huddle session.

### Projects

- Projects are private by default.
- Use them for kanban tasks, notes, links, uploaded files, and project-scoped collaboration.
- Launch a project huddle when the work needs a live conversation.

### Ticket Triage

- Create a ticket when a problem or request needs review.
- Tickets can be approved into projects or denied with decision context.

## 5. Use campaign tools

### Social

- Draft outbound posts.
- Choose channels.
- Schedule publishing jobs.
- Track the per-channel publish state.

### Voice

- Import leads by CSV.
- Create a voice campaign batch.
- Track transcript summaries and call outcomes as the runtime layer matures.

### SMS

- Import contacts by CSV.
- Set a base message template.
- Track delivery states and future reply capture.

## 6. When to use the Local Listener

Use the Local Listener when the work must happen on a trusted local machine, such as:

- coding in a real repository
- running a local CLI workflow
- producing artifacts with local file access
- continuing a cloud-planned task in Gemini or Codex locally

## 7. Best practices

- Share threads when multiple people need the same context.
- Use projects when the work needs structured tracking, artifacts, or private access control.
- Use tickets before projects when something needs explicit triage.
- Use huddles for real-time collaboration, not for long-term records alone.
- Keep integrations configured in the Integration Wizard so campaign tools stay ready.
